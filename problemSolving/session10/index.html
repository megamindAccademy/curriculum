How to Read a Problem üìñ

On platforms like Codeforces or HackerRank, every problem has a specific structure. You must read it like a detective! üïµÔ∏è‚Äç‚ôÇÔ∏è

‚è≥

1. Time & Memory Limit

The rules! E.g., "Time limit: 1 second". Your code must be fast!

üìù

2. Problem Statement

The story or the puzzle you need to solve. Read it carefully!

üì•

3. Input

What data the Judge will give to your program (e.g., "A single integer n").

üì§

4. Output

What your program MUST print (e.g., "Print YES or NO").

üß™

5. Examples (Test Cases)

Samples to test your code before submitting. If input is 8, output should be YES.

The Judge's Verdicts ‚öñÔ∏è

When you submit your code, an automated "Judge" tests it. Here is what it can tell you:

‚úÖ

Accepted (AC)

Perfect! Your code gave the correct answer for ALL hidden test cases!

‚ùå

Wrong Answer (WA)

Your code ran, but printed the wrong result on one of the tests (e.g., Wrong answer on test 1).

‚è≥

Time Limit Exceeded (TLE)

Your code is too slow! Usually means you have an infinite loop or bad logic.

üí•

Compilation Error

Syntax error! The Judge couldn't even run your code because of a typo or bad grammar.
